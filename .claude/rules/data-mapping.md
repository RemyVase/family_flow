# Data mapping — Firestore ↔ models

Repositories only make calls and must not contain logic — but something has to
convert `DocumentSnapshot` ↔ Freezed model. This is where it happens, and how.

## The rule

- Serialization lives **on the model**: every persisted model has
  `fromJson` / `toJson` generated by Freezed + `json_serializable`
  (see @.claude/rules/models-freezed.md).
- The repository uses Firestore's **`withConverter`** so reads/writes are
  already typed. The repository **returns typed models** (or streams/lists of
  models) — **never a raw `Map`, never a `DocumentSnapshot`** leaks up to
  services or controllers.
- Mapping is pure (de)serialization only. No business transformation, no
  computed/derived fields in the repository — that belongs to a service or the
  controller.

## Model side

```dart
@freezed
class User with _$User {
  const factory User({
    required String id,
    required String name,
  }) = _User;

  factory User.fromJson(Map<String, dynamic> json) => _$UserFromJson(json);
}
```

## Repository side — typed via `withConverter`

```dart
CollectionReference<User> get _users =>
    _firestore.collection('users').withConverter<User>(
          fromFirestore: (snap, _) => User.fromJson({...snap.data()!, 'id': snap.id}),
          toFirestore: (user, _) => user.toJson()..remove('id'),
        );

Future<User?> fetchUser(String id) async {
  final snap = await _users.doc(id).get();
  return snap.data(); // already a User
}

Stream<List<User>> watchUsers() =>
    _users.snapshots().map((q) => q.docs.map((d) => d.data()).toList());
```

## Conventions

- The document `id` is not stored as a field; inject it from `snap.id` on read
  and strip it on write (as above).
- Keep Firestore field names aligned with the model's JSON keys; use
  `@JsonKey(name: ...)` when they must differ.
- Timestamps: convert `Timestamp` ↔ `DateTime` in a `JsonConverter`, applied on
  the model, so it stays out of the repository body.
